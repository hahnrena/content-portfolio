# Rena Hahn — Portfolio System Framework

## Core Principle

A portfolio is not a showcase.

It is a **system that demonstrates how you think, design, and build at scale.**

---

## Portfolio Structure

Every case study MUST follow this structure:

### 1. Context (Problem Framing)
- What system existed before?
- What was broken, slow, or unclear?
- Who was affected?

Avoid vague statements. Must be concrete.

---

### 2. System Design (Not Features)

You must show:
- Information architecture
- Data flow
- System structure
- Decision hierarchy

This is the most important section.

---

### 3. Your Intervention

Clearly define:
- What you designed
- What you built
- What you changed structurally

Avoid listing tasks.

---

### 4. Execution

Include:
- Components built
- Tools used
- Architecture decisions
- Tradeoffs

---

### 5. Impact (Must be quantified)

Every case study must include:

- Time saved
- Efficiency gained
- Scale (# users, teams, systems)
- Reduction in friction or errors

If exact numbers are unknown:
→ use conservative estimates and clearly label them as such

---

## Case Study Types (ONLY THESE 3)

### 1. Knowledge System Case Study
Focus:
- Documentation systems
- Information architecture
- Internal knowledge workflows

---

### 2. AI Workflow Case Study
Focus:
- LLM pipelines
- Prompt systems
- Automation of structured outputs

---

### 3. Product Engineering Case Study
Focus:
- Scalable frontend systems
- Component architecture
- Multi-client deployment systems

---

## Quality Bar

A case study is INVALID if:

- It describes UI without system design
- It lacks measurable impact
- It is a list of tasks instead of decisions
- It does not explain structure or reasoning

---

## Narrative Rule

Every case study must reinforce:

"I build systems that make complex workflows and information usable at scale through structured design and engineering."

---

## Portfolio Anti-patterns

DO NOT:
- Show pretty UI without system logic
- Over-focus on visuals
- List features without architecture
- Ignore scalability considerations
